1. Introduction

A service worker is capable of fetching and caching assets, the size of which is restricted only by origin storage. However, if the user navigates away from the site or closes the browser, the service worker is likely to be killed. This can happen even if thereâs a pending promise passed to waitUntil(); if it hasnât resolved within a few minutes the browser may consider it an abuse of service worker and kill the process.

This makes it difficult to download and cache large assets such as podcasts and movies, and upload video and images. Even if the service worker isnât killed, having to keep the service worker in memory during this potentially long operation is wasteful.

This specification aims to:

  • Allow fetches to continue even if the user closes all windows & worker to the origin.

  • Allow a single job to involve many requests, as defined by the app.

  • Allow the browser/OS to show UI to indicate the progress of the fetch, and allow the user to pause/abort.

  • Allow the browser/OS to deal with poor connectivity by pausing/resuming the download/upload (may be tricky with uploads, as ranged uploads arenât standardized).

  • Allow the app to react to success/failure of the background fetch group, perhaps by caching the results.

  • Allow access to background-fetched resources as they fetch.

  • Allow the app to display progress information about a background fetch.

  • Allow the app to suggest which connection types the fetch should be restricted to.

2. Infrastructure

The background fetch task source is a task source. All queued tasks in this specification use the background fetch task source unless otherwise noted.

2.1. Extensions to service worker registration

A service worker registration additionally has:

  • An associated active background fetches (a map), where each key is a DOMString, and each item is a background fetch. It is initially an empty map.

  • An associated active background fetches edit queue (a parallel queue), initially the result of starting a new parallel queue.

To determine whether an active background fetches contains bgFetch (a background fetch), run these steps:

  1. Let id be bgFetchâs id.

  2. If activeBgFetches[id] does not exist, then return false.

  3. If activeBgFetches[id] does not equal bgFetch, then return false.

  4. Return true.

2.2. Background fetch

A background fetch consists of:

  • An id (a DOMString).

  • Records (a list of background fetch records).

  • A title (a DOMString).

  • A download total (a number).

  • An upload total (a number).

  • A last reported download (a number), initially 0.

  • Icons (a list of IconDefinitions).

  • A service worker registration (a service worker registration).

  • A progress handling queue (a parallel queue), initially the result of starting a new parallel queue.

  • A paused flag, initially unset.

2.3. Background fetch record

A background fetch record consists of:

  • A request (a request).

  • A response (a response), initially null.

  • A response body (a byte sequence), initially empty.

3.2. Report progress for background fetch

To report progress for background fetch bgFetch (a background fetch), enqueue the following steps to bgFetchâs progress handling queue:

  1. Let downloaded be 0.

  2. For each record in bgFetchâs records, increment downloaded by the number of bytes in response body.

  3. If downloaded is equal to bgFetchâs last reported download, then abort these steps.

  4. Set bgFetchâs last reported download to downloaded.

  5. For each environment settings object env whose origin is equal to bgFetchâs service worker registration's scope URL's origin, queue a task on envâs responsible event loop to run these steps:

    1. Let bgFetchRegistration be the instance of BackgroundFetchRegistration within this environment whose background fetch is equal to bgFetch, or null if none exists.

      Note: There will be at most one per environment, because of the get a BackgroundFetchRegistration instance algorithm.

    2. If bgFetchRegistration is not null, then:

      1. Set downloaded to downloaded.

      2. fire an event named "progress" at bgFetchRegistration.

3.3. Get a BackgroundFetchRegistration instance

To get a BackgroundFetchRegistration instance for an instancesMap (a BackgroundFetchRegistration instances) and a bgFetch (a background fetch), run these steps:

  1. If instancesMap[bgFetch] exists, then return instancesMap[bgFetch].

  2. Let instance be a new BackgroundFetchRegistration in the parent BackgroundFetchManager's relevant Realm, and set itâs background fetch to bgFetch.

  3. Set instancesMap[bgFetch] to instance.

  4. Return instance.

Note: This is to ensure the same instance is returned for a given background fetch throughout the life of a BackgroundFetchManager. Itâs okay for browsers to optimise this, as long as thereâs no way to tell that more than one instance has been created for a given background fetch (e.g through equality, expandos, or weakly-associated data).

3.4. Validate a partial response

To validate a partial response for a request (a request), a partialResponse (a response), and an optional previousPartialResponse (a response or null, null unless otherwise specified), run these steps:

  1. Assert: requestâs header list contains `Range`.

  2. Assert: partialResponseâs status is 206.

  3. Let rangeValue be the value of the first header whose name is a byte-case-insensitive match for `Range` in requestâs header list.

  4. If parsing rangeValue per single byte range fails, then return invalid.

  5. Let responseFirstBytePos, responseLastBytePos, and responseCompleteLength be the result of extracting content-range values from partialResponse. If this fails, then return invalid.

  6. If requestFirstBytePos does not equal responseFirstBytePos, then return invalid.

  7. If previousPartialResponse is given, then:

    1. If previousPartialResponseâs header list contains `ETag`, and header equality for `ETag`, previousPartialResponse, and partialResponse is false, then return invalid.

    2. If previousPartialResponseâs header list contains `Last-Modified`, and header equality for `Last-Modified`, previousPartialResponse, and partialResponse is false, then return invalid.

    3. If responseCompleteLength is not null, then:

      1. Let previousResponseFirstBytePos, previousResponseLastBytePos, and previousResponseCompleteLength be the result of extracting content-range values from previousPartialResponse. If this fails, then return invalid.

      2. If responseCompleteLength does not equal previousResponseCompleteLength, then return invalid.

  8. Return valid.

3.5. Extract content-range values

To extract content-range values from a response (a response), run these steps:

  1. If responseâs header list does not contain `Content-Range`, then return failure.

  2. Let contentRangeValue be the value of the first header whose name is a byte-case-insensitive match for `Content-Range` in responseâs header list.

  3. If parsing contentRangeValue per single byte content-range fails, then return failure.

  4. Let firstBytePos be the portion of contentRangeValue named first-byte-pos when parsed as single byte content-range, parsed as an integer.

  5. Let lastBytePos be the portion of contentRangeValue named last-byte-pos when parsed as single byte content-range, parsed as an integer.

  6. Let completeLength be the portion of contentRangeValue named complete-length when parsed as single byte content-range.

  7. If completeLength is "*", then set completeLength to null, otherwise set completeLength to completeLength parsed as an integer.

  8. Return firstBytePos, lastBytePos, and completeLength.

3.6. Header equality

To determine header equality for a headerName (a byte sequence), in response1 (a response), and response2 (a response), run these steps:

  1. If response1âs header list does not contain headerName, and response2âs header list does not contain headerName, then return true.

  2. If response1âs header list contains headerName, and response2âs header list contains headerName, then:

    1. Let response1Value be the value of the first header whose name is a byte-case-insensitive match for headerName in response1âs header list.

    2. Let response2Value be the value of the first header whose name is a byte-case-insensitive match for headerName in response2âs header list.

    3. If response1Value is identical to response2Value, then return true, otherwise return false.

  3. Return false.

4. Range header syntax

The following is ABNF for a single byte range:

"bytes=" first-byte-pos "-" [ last-byte-pos ]
first-byte-pos = 1*DIGIT
last-byte-pos  = 1*DIGIT

The following is ABNF for a single byte content-range:

"bytes=" first-byte-pos "-" last-byte-pos "/" complete-length
complete-length = ( 1*DIGIT / "*" )

5.3.1. fetch()

The fetch(id, requests, options) method, when invoked, run these steps:

  1. Let registration be the context object's service worker registration.

  2. Let bgFetchInstances be context object's BackgroundFetchRegistration instances.

  3. Let records be a new list.

  4. Let uploadTotal be 0.

  5. If requests is a RequestInfo, set requests to Â« requests Â».

  6. If requests is empty, then return a promise rejected with a TypeError.

  7. For each request of requests:

    1. Let internalRequest be the request of the result of invoking the Request constructor with request.

    2. If internalRequestâs mode is "no-cors", then return a promise rejected with a TypeError.

    3. Set internalRequestâs client to null.

    4. Let record be a new background fetch record.

    5. Set recordâs request to internalReqeust.

  8. Let promise be a new promise.

  9. Enqueue the following steps to registrationâs active background fetches edit queue:

    1. Let bgFetchMap be registrationâs active background fetches.

    2. If registrationâs active worker is null, then reject promise with a TypeError and abort these steps.

    3. If bgFetchMap[id] exists, reject promise with a TypeError and abort these steps.

    4. Let requestBodiesRemaining be the size of requests.

    5. Let requestReadAborted be false.

    6. For each request of requests:

      1. If requestâs body is null, then continue.

      2. Let stream be requestâs body's stream.

        1. Run these steps but abort when requestReadAborted is true:

        4. Decrement requestBodiesRemaining by 1.

    9. If requestReadAborted is true, then reject promise with a TypeError and abort these steps.

      id

      id.

      records

      records.

      download total

      optionsâ downloadTotal member.

      upload total

      uploadTotal.

      icons

6. Privacy and bandwidth usage

Itâs the recommendation of this standard that user agents make background fetch operations highly visible and easily abortable rather than asking permission up front. However, given that background fetches can be paused and retried, a user agent that immediately paused a background fetch and user interaction to resume would be compliant.

The origin of a background fetch should have equal or great visibility to the developer-provided information, such as a the icons and title.

A user agent may pause background downloads depending on network conditions or battery state. For example, a user agent may download automatically when connected to WiFi, but require user confirmation to download using mobile data. If an implementation has this restriction, the user should be able to persist granting mobile data use for the origin.

If the user agent provides a way for a user to abort a background fetch operation, it should terminate all fetches associated with the background fetch's records, with the abort flag set.

A background fetch exposes the userâs external IPs to the server throughout the duration of the background fetch. An attacker could use an extremely slow background fetch as a way to persistently track the user once theyâve left the site. Good visibility into the progress of the background fetch will help the user identify unwanted use.
